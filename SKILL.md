# BOSS直聘消息与简历处理技能

## 目标

- 清空待处理消息并推进候选人状态。
- 使用已存储岗位规则做一致评估，不临时改标准。
- 执行“先判定再下载”，减少无效下载和无效上传。
- 批量导入飞书时稳定处理三种状态：正常、缺必填项、重复人才。

## 强规则（必须执行）

1. 默认只处理“今天收到/今天需处理”的简历，除非用户明确放宽日期范围。
2. 规则读取优先级固定为：`effective_rules > jobs[].draft_overrides > global_rule_defaults`。
3. 必须先判定是否通过岗位规则，再下载简历。
4. 在 BOSS 不通过操作中，统一选择 `其他原因`，避免触发额外消息。
5. 评估标准优先读取 `bosszhibin-auto-recruiter` 的岗位生效规则（effective rules）；不得随意放宽硬性条件。
6. 硬性条件（学历/年限/核心技能）任一不通过即判定不通过；仅在用户明确特批时可保留。
7. 用户特批保留时，必须将决策写入缓存：`override_flag=true` + `override_reason`。
8. 进入飞书批量导入后，先处理 `缺必填项`，再处理 `重复人才`，最后提交。
9. 飞书页面操作前先清除遮挡弹窗（反馈/引导/浮层），再点击核心按钮。
10. 下载策略默认“直接落盘到目标目录”；仅在浏览器限制下使用临时目录中转并立即迁移。

## 输入前置

执行前确认：

- 已登录 BOSS 直聘企业端。
- 已登录飞书招聘。
- 已知当前职位及其生效规则（来自岗位缓存/用户确认规则）。
- 飞书批量导入来源为 `第三方招聘网站 -> BOSS直聘`。
- 已确认简历目标目录：`/Users/blacklake/Downloads/new/downloaded_resumes/`。
- 默认请求简历话术可用：

```text
可以的话，要不发我一份简历，我们根据简历来简单聊聊？
```

## 执行前清障（必做）

进入飞书批量新建页面后，先执行一次清障：

1. 关闭右下角 `反馈` 弹窗（优先点 `取消`，不提交反馈）。
2. 关闭新手引导、客服浮层、遮挡上传区域的悬浮窗。
3. 若遮挡未消失，按 `Esc` 后重试点击目标按钮。
4. 仍无法点击时记录 `ERROR_UI_BLOCKED`，刷新页面再重试一次。

## 触发命令

| 命令 | 动作 |
|---|---|
| `处理今天简历` | 只处理今天范围内候选人，从判定到下载 |
| `处理消息` | 扫描会话并请求/接收简历 |
| `处理简历` | 按岗位规则完成评估并给出通过结论 |
| `批量上传飞书` | 执行飞书批量新建并处理缺项/重复 |
| `批量处理简历` | 从 BOSS 到飞书整批闭环处理 |

## 执行原则

1. 先识别状态，再执行动作。
2. 每位候选人一次只推进一个状态。
3. 任何会触达候选人的动作都遵循最小触达原则。
4. 使用 `candidate_id + job_id` 幂等键，已完成步骤默认跳过。
5. 批量模式每处理 5 人输出一次进度。
6. 每次点击前确认页面上下文（当前 Tab、当前弹窗、当前候选人）与目标动作一致，避免误操作。

## 状态机

```text
NEW_MESSAGE
  -> REQUESTED_RESUME
  -> RESUME_RECEIVED
  -> PRECHECK_PASSED / PRECHECK_REJECTED / PRECHECK_OVERRIDDEN
  -> DOWNLOADED
  -> FEISHU_IMPORTING
  -> FEISHU_FIXED_REQUIRED_FIELDS
  -> FEISHU_DUPLICATE_UPDATED
  -> DONE

任意状态可转：REJECTED / ON_HOLD / ERROR
```

## 主流程

### 1) BOSS 消息扫描与分类

进入消息列表后，按以下顺序收集待处理会话：

- 候选人已回复但我方未回复。
- 候选人主动投递并等待处理。
- 候选人发起“发送简历”请求待同意。
- 会话中已有简历但尚未评估。

### 2) 会话处理

对每个会话执行：

1. 打开会话并读取最近消息。
2. 判断分支：
- 已有简历：跳转到“简历评估”。
- 无简历且候选人有意向：发送标准请求简历话术。
- 出现“对方想发送简历给你”：点击“同意”。
- 候选人明确拒绝或不匹配：标记 `REJECTED` 并结束该会话。
3. 写入缓存并返回消息列表。

### 3) 先判定再下载（关键）

在 BOSS 端先完成预判：

- 先按优先级读取规则：`effective_rules > jobs[].draft_overrides > global_rule_defaults`。
- 使用规则判断硬性条件是否通过。
- 未通过：标记 `PRECHECK_REJECTED`，不下载简历，直接进入不通过处理。
- 用户明确特批保留：标记 `PRECHECK_OVERRIDDEN`，写入特批原因后允许下载。
- 通过：标记 `PRECHECK_PASSED`，允许下载并进入飞书上传候选池。

### 4) 简历评估规则

按岗位规则输出 100 分制结果，先硬性条件，后加权评分。

#### 4.1 硬性条件（必须先判）

- 学历门槛（例如 985/QS100/本科及以上）。
- 工作年限门槛。
- 核心技能门槛（例如 LLM/RAG/PyTorch）。

任一不通过即结论 `不合适`。

#### 4.2 权重评分（总分 100）

- 教育背景：25
- 工作经历与公司背景：25
- 核心技能匹配：30
- 领域经验与项目产出：20

#### 4.3 输出模板

```text
简历评估
候选人：<姓名>
职位：<职位名称>
总分：<0-100>
结论：<合适/不合适/待定>

硬性条件：
- 学历：<通过/不通过>
- 年限：<通过/不通过>
- 核心技能：<通过/不通过>

关键证据：
- <证据1>
- <证据2>

风险点：
- <风险1>

建议动作：<上传飞书/保留观察/淘汰>
请用户确认：合适 / 不合适 / 待定
```

### 5) 不通过处理（BOSS）

当结论为不通过：

1. 若当前在简历预览层，先返回会话主面板再执行动作。
2. 点击 `不合适`，理由统一选择 `其他原因`。
3. 记录原因与证据，不发送额外说明消息。
4. 状态写入 `REJECTED`。

### 6) 下载与整理

仅对 `PRECHECK_PASSED` 或 `PRECHECK_OVERRIDDEN` 的候选人下载简历：

1. 优先直接下载到 `/Users/blacklake/Downloads/new/downloaded_resumes/`。
2. 若浏览器只能落到默认下载目录，下载后立即迁移到目标目录并校验文件存在。
3. 校验失败时标记 `ERROR_DOWNLOAD_MISSING`，不进入飞书上传。
4. 文件名保留原始姓名信息；重名时追加短后缀（如 `_2`）避免覆盖。

### 7) 飞书批量上传

1. 进入飞书候选人列表后，先执行一次“执行前清障”。
2. 点击 `+新建 -> 批量新建人才`。
3. 来源选择 `第三方招聘网站 -> BOSS直聘`。
4. 上传本次通过或特批保留且已下载的简历。
5. 校验已选择文件数是否等于待上传数，不一致则暂停并排查漏传。
6. 仅在页面出现“成功上传 N 份简历”后点击 `下一步` 进入逐条检查页。

### 8) 飞书逐条处理（三种状态）

#### 8.1 正常简历

- 不修改，保持待提交。

#### 8.2 缺必填项

按以下步骤处理每一条：

1. 点击该条 `缺必填项` 简历。
2. 在右侧表单区域下拉，定位红色校验提示（如 `请填写完整时间`）。
3. 优先修复教育经历起止时间缺失；若结束时间缺失且为在读，点击 `目前年月`。
4. 若仍缺失，再手动补齐为当前年月（`YYYY-MM`）。
5. 修复后确认左侧该条 `缺必填项` 标签消失。

#### 8.3 重复人才

按以下步骤处理：

1. 点击 `重复人才` 简历。
2. 点击右侧 `发现重复人才` 链接，进入已有档案核对身份信息（姓名/手机/邮箱）。
3. 至少两项关键信息一致（如姓名+手机，或姓名+邮箱）才按重复更新处理。
4. 若信息冲突，标记 `ERROR_DUPLICATE_CONFLICT` 并置为 `ON_HOLD`，不提交该条。
5. 返回批量页后继续提交。
6. 在确认弹窗中看到 `X 个重复人才的简历将更新` 时点击 `确定`。
7. 结果应为更新已有人才，不新建重复档案。

### 9) 用户确认门禁（可选）

如用户要求“先确认再上传”，评估后先停在确认点：

- `合适`：进入飞书上传流程。
- `不合适`：按“不通过处理”执行。
- `待定`：标记 `ON_HOLD`，不执行上传。

### 10) 提交与完成

当满足以下条件后再提交：

- `缺必填项=0`
- 重复人才已核对且无 `ERROR_DUPLICATE_CONFLICT`
- 上传成功数 = 待上传数（通过+特批保留）

执行：

1. 点击 `提交`。
2. 确认弹窗点击 `确定`。
3. 返回候选人列表即视为本批完成。
4. 写入缓存状态：`DONE`。

## 批量模式

启动命令：`批量处理简历`。

批量执行策略：

- 优先级：今天已获取简历 > 今天待同意收简历 > 今天待请求简历。
- 节奏：逐个处理，不并发点击。
- 汇报：每 5 人输出一次进度；结束后输出总汇总。
- 中止条件：
- 连续 3 次页面异常或按钮不可点击。
- 飞书插件连续 2 次无响应。
- 用户发送“停止”。

## 缓存与幂等

缓存文件：`./bosszhibin_resume_cache.json`

建议结构：

```json
{
  "schema_version": "1.2",
  "updated_at": "2026-02-26T00:00:00Z",
  "candidates": {
    "<candidate_id>#<job_id>": {
      "status": "EVALUATED",
      "score": 82,
      "decision": "ON_HOLD",
      "decision_source": "effective_rules",
      "override_flag": false,
      "override_reason": "",
      "rule_chain": [
        "effective_rules",
        "jobs[].draft_overrides",
        "global_rule_defaults"
      ],
      "last_action": "resume_review",
      "last_updated": "2026-02-26T00:00:00Z"
    }
  }
}
```

规则：

- 写入前先读缓存，命中同一幂等键且目标状态已完成则跳过。
- 状态回退只能在用户明确指令下执行。
- 用户特批通过必须落缓存：`decision_source=manual_override`、`override_flag=true`、`override_reason` 不为空。

## 异常处理

- 简历无法识别：提示用户手动查看并补充关键信息后继续。
- 飞书插件无响应：刷新页面后重试 2 次，仍失败则标记 `ERROR` 并暂停该候选人。
- 职位列表为空：提示检查飞书权限或职位配置，不盲目上传。
- 上传失败：记录错误信息，提供“重试上传”单步操作。
- 飞书右下角“反馈”弹窗遮挡操作：先点 `取消` 或按 `Esc` 关闭，不提交反馈弹窗。
- 点击被遮挡（Click intercepted）：先执行清障，再刷新重试 1 次，仍失败标记 `ERROR_UI_BLOCKED`。
- 下载后目标目录无文件：执行默认下载目录搜寻与迁移；仍缺失则标记 `ERROR_DOWNLOAD_MISSING`。

## 与 bosszhibin-auto-recruiter 协作边界

- `bosszhibin-auto-recruiter`：负责职位解析、规则缓存、主动打招呼与初筛。
- 本技能：负责消息收口、严格判定、下载、飞书批量导入与修复提交。
- 交接信号：候选人进入“可收简历/已收简历/已获取简历”状态后切换到本技能。
